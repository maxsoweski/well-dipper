# System Contracts

Design principles and invariants that all code, reviews, and feature work must respect.

## 1. Environmental Contract: Procedural Galaxy Model

**All spatial content in the game world is derived from the procedural galaxy model.**

The galaxy is generated by `GalacticMap.js` using a density-based model with spiral arms, a central bar, thin/thick disk populations, bulge, and halo. Every star in the sky, every star in the nav computer, and every warp destination comes from the `HashGridStarfield` — a deterministic hash grid that maps galactic coordinates to star properties.

### What this means:

- **Every warp destination is a real hash grid star.** The DestinationPicker may roll categories (star-system, nebula, cluster) but the actual destination must resolve to a hash grid position. If a feature type is rolled but no matching real feature exists nearby, the destination falls back to a star system — NOT to a legacy fake generator.

- **No fabricated spatial objects.** The `NavigableNebulaGenerator` and `NavigableClusterGenerator` are LEGACY code from a pre-galaxy branch. They create fake tourrable environments that don't exist in the galaxy model. They must NOT be reachable from the normal warp/screensaver loop. They may still be used for the object gallery (debug/showcase mode) but never for gameplay destinations.

- **Nebulae and clusters are sky features, not destinations.** On the realistic galaxy branch, nebulae are rendered as sky-layer features (ProceduralGlowLayer, SkyFeatureLayer) — background elements you see or fly through. They are NOT discrete objects you warp to and tour. The `RealFeatureCatalog` provides real globular cluster positions from astronomical data, but these are rendered as sky features, not navigable systems.

- **The title screen nebula is a visual prop, not a gameplay location.** It's spawned for aesthetics using NebulaGenerator (the billboard renderer), not NavigableNebulaGenerator. The player never "visits" it — they warp away to their first real star system.

### Invariants:

1. `onPrepareSystem()` must always produce a system from: hash grid star → StarSystemGenerator, OR KnownSystems override (Sol, etc.), OR a GalacticMap feature position, OR an external galaxy (Easter egg). Never from NavigableNebulaGenerator, NavigableClusterGenerator, or ClusterGenerator for warp destinations.
2. `autoSelectWarpTarget()` must always select a visible hash grid star. If it fails, abort the warp — don't fall back to fake generators or directionless warps.
3. `beginWarpTurn()` must never fire without a real warp target. If `warpTarget.direction` is null, it calls `autoSelectWarpTarget()` as a safety net. If that also fails, the warp is aborted.
4. The nav computer's column/system views only show hash grid stars. No fabricated entries.
5. The sky renderer's features (nebulae, clusters, dust) come from the GalacticMap density model and RealFeatureCatalog. They are rendering elements, not gameplay destinations.

### Approved uses of legacy generators:

- **Title screen nebula:** `NebulaGenerator` creates a cosmetic backdrop. It is NOT a gameplay destination — the player warps away to a real hash grid star.
- **External galaxy Easter egg:** `GalaxyGenerator` renders distant galaxies the player can warp to. These are outside the Milky Way model by definition. A "you've gone too far" message should display on arrival.
- **Debug/gallery mode:** Legacy generators may be used for debug spawns and the object gallery. These are developer tools, not gameplay paths.
- **No other use is permitted.** If a warp destination can't be resolved to a hash grid star or real catalog feature, the warp must fall back to `star-system` or abort.

## 2. Camera Ownership Contract

The camera is driven by one system at a time:

| Owner | When | Controls |
|-------|------|----------|
| ShipCameraSystem | Manual play (orbit, WASD flight, zoom) | `cameraController.bypassed = false` |
| FlythroughCamera | Autopilot tour OR manual COMMIT BURN transit | `flythrough.active = true`, `cameraController.bypassed = true` |
| Warp system | During warp turn + warp effect phases | `cameraController.bypassed = true`, direct camera manipulation |

**Invariant:** `cameraController.bypassed` must be true whenever FlythroughCamera or warp is driving. It must be restored to false when manual control resumes. The flythrough's `travelComplete` handler restores it for manual burns.

## 3. Nav Computer Contract

The nav computer is scene-agnostic. It knows:
- Galactic coordinates (kpc)
- Star seeds, spectral types, names
- Planet/moon indices
- Player position and current body index

It does NOT know:
- THREE.js objects, meshes, cameras
- autoNav, flythrough, warp effect state
- Scene graph structure

**Communication is through the action contract:**
```
NavAction = { type: 'burn'|'warp', target, starIndex, planetIndex, moonIndex, star }
```

NavComputer builds it. main.js dispatches it via `dispatchNavAction()`. No cross-callbacks.
